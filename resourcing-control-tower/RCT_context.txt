Resource Assignation System (RAS) - Application Context
=======================================================

Last updated: 2026-05-14


Purpose
-------
Resource Assignation System (RAS), formerly Resourcing Control Tower, is an
offline Windows desktop application for leadership review of resource
assignation, billing, assignment end dates, PM ownership, PP financial KPI
movement, and RFS forecast coverage.

The app is local-only:
- No backend.
- No authentication.
- No database.
- Excel files are the source of truth.


Technology
----------
Frontend:
- React with Vite.
- MUI DataGrid.
- Recharts.

Desktop:
- Electron.
- Electron main process handles local filesystem access, Excel loading, file
  selection, and Excel export.
- Electron preload exposes safe IPC APIs with contextBridge.

Excel:
- SheetJS/xlsx.
- Supports `.xlsx`, `.xls`, `.xlsm`, and `.xlsb` where enabled by module.

Packaging:
- electron-builder.
- Windows NSIS installer.
- Installer output:
  `release/Resource Assignation System (RAS)-1.0.0-Setup.exe`


Build Commands
--------------
- `npm.cmd run build`
- `npm.cmd run dist`

Recently verified:
- Both commands completed successfully after the RAS naming and PP period
  comparison updates.

Packaging notes:
- `vite.config.js` uses `base: './'` so packaged Electron can load renderer
  assets from `file://`.
- Electron-builder currently uses the default Electron icon because no custom
  icon is configured.

--------------------

- `selectedWorkbookPath`
- `selectedPpWorkbookPath`
- `selectedRfsWorkbookPath`

This allows refresh to reload the same selected workbook after file selection.

Electron user-data `comments.json` stores editable comments:
- `resource`
  - RAS PM comments / PMO comments keyed by employee, customer, project, and
    assignment end date.
- `pp`

Comment persistence rules:
- Comments are local to the Windows user profile.
- Source Excel workbooks are not modified.
- Reloading the app or refreshing Excel reapplies saved comments where keys
  match the current workbook rows.

Billing / WBS Rules
-------------------
Billing Status:
- `B` = `Billable`
- `C` = `Contractual Shadow`
- `N` = `Non Billable`

Y-Code / C-Code:
- Derived from the first character of `WBS Code`.
- Starts with `Y` => `Y-Code`.
- Starts with `C` => `C-Code`.
- Otherwise => `Other`.


FTE Counting Rule
-----------------
All leadership counts use FTE-weighted values, not unique employee count.

Example:
- If a resource has FTE `0.25`, it contributes `0.25`.

Fallback:
- If FTE is missing, FTE falls back to `allocationPct / 100`.

Allocation normalization:
- `1` becomes `100`.
- `0.25` becomes `25`.


Derived Resourcing Logic
------------------------
End-date:
- `daysToEnd` = assignment end date minus current date.
- `endRiskFlag` = true when `daysToEnd` is between 0 and 30.

Bench:
- True when Billing Status is `Non Billable`, or WBS type is `Y-Code` or
  `C-Code`.

Fresher:
- True when `Fresher/FCL /Lateral` starts with `Fresher`.

Utilization:
- `(Total FTE - Non Billable FTE) / Total FTE * 100`

PP KPI Order
------------
The detailed KPI comparison uses this order:
1. Total Revenue
2. Revised DRC
3. Revised GM
4. GM%
5. Total Project Expenses
6. Realization
7. Total ARC
8. Total GFTE
9. Total AFTE
10. Total BFTE
11. Unbilled
12. GFTE Utili %
13. AFTE Utili %


PP KPI Formulas
---------------
Direct:
- Total Revenue
- Revised DRC
- Revised GM
- Total Project Expenses
- Total GFTE
- Total AFTE
- Total BFTE

Calculated:
- Realization = Total Revenue / Total AFTE
- Total ARC = Revised DRC / Total GFTE
- GM% = Revised GM / Total Revenue
- GFTE Utili % = Total BFTE / Total GFTE
- AFTE Utili % = Total BFTE / Total AFTE
- Unbilled = Total AFTE - Total BFTE

Division by zero returns 0.

Detailed KPI Comparison by Customer
-----------------------------------
Behavior:
- Customer results are grouped under collapsible cards.
- Each card title is the customer/customer group name.
- First customer card is expanded by default.
- Customer column is not shown inside the KPI grid.
- Grid rows are compact.

Columns:
- KPI.
- Each selected comparison period.
- 3-Month Avg in Month mode.
- Comparison Avg in Qtr/Fiscal modes.
- Current period.
- Change.
- RFS Value.
- Trend.
- Action Flag.
- PM Remarks.

PM Remarks:
- Editable in the grid.
- Persist locally in Electron `userData/comments.json`.
- Reloading PP Analysis reapplies saved remarks when period type, current
  period, customer, and KPI match.
- Not written back to source Excel.

Action Flag:
- Derived through a configurable PP rule engine.
- Rules are evaluated in the UI against the current filtered PP scope and, for
  Total Revenue, the matching RFS customer-group value where available.
- Action flag colors are configurable.

Current action rule behavior:
- Total Revenue:
  - Compare current period value against corresponding RFS value.
  - If PP revenue is lower: `Missing Revenue or UBR or RT based on RFS`.
  - Double-click opens Total Revenue details by `Cost Element_6`.
- Revised DRC:
  - Indicative DRC = previous period Total ARC * current period Total GFTE.
  - Above indicative DRC flags possible added high-cost resources.
  - Below indicative DRC flags possible released high-cost resources.
- Revised GM and GM%:
  - Positive movement flags improved GM.
  - Negative movement flags decreased GM.
- Total Project Expenses:
  - Double-click opens expense details by `Cost Element_6`.
  - Deviation above configured threshold, default 10%, is highlighted.
- Total GFTE, Total AFTE, Total BFTE:
  - Positive movement flags ramp up.
  - Negative movement flags ramp down.
- Realization:
  - Default improvement threshold is current value more than 10% above
    comparison average.
  - Decline flags missed revenue or unbilled resources.
- Total ARC:
  - Positive movement flags increased cost.
  - Negative movement flags decreased cost.
- GFTE Utili % and AFTE Utili %:
  - Default red below 80.
  - Default amber from 80 to below 90.
  - Default green at 90 and above.
- Unbilled:
  - Default threshold is >=10% of Total GFTE.


PP Color Coding
---------------
Applies to period/current/average/change numeric cells:
- Negative numbers are red by default.
- Negative number color is configurable in PP Rule Configuration.

GM% thresholds:
- GM% <=25: red by default.
- GM% 26-35: amber by default.
- GM% >=36: green by default.
- Thresholds are configurable.

Note:
- Negative number red takes priority over threshold coloring.

RFS and PP Relationship
-----------------------
RFS Tracker is intended to support PP Analysis action flags.

Current status:
- RFS summary is calculated and filtered independently.
- PP Analysis now reads RFS summary values for Total Revenue action flag
  comparison when customer group names match.
- PP month comparison uses the matching RFS month field from the loaded
  quarter summaries.
- PP quarter comparison uses the matching RFS quarter total.
- PP fiscal comparison sums matching customer-group RFS values across AMJ,
  JAS, OND, and JFM.

Potential future refinement:
- Add deeper PP/RFS combined action models, such as:
  - Low GM% with strong RFS pipeline.
  - Revenue decline with weak RFS.
  - High unbilled with low future RFS coverage.

Known Limitations
-----------------
- Source Excel files are not modified.
- Editable RAS comments and PP remarks persist locally, but they are not yet
  synced across users or written back to source workbooks.
- PP and RFS are partially connected for Total Revenue RFS comparison but are
  not yet merged into a single combined action model.
- RFS supports AMJ, JAS, OND, and JFM source/output sheet selection.
- PP supports Month, Qtr, and Fiscal period comparison using workbook fields.
- No backend storage.
- No authentication or role-based access.
- No Power BI embedding yet.
- No desktop notification alerts yet.
